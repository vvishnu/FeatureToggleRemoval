# 🔧 Universal Feature Toggle Removal Guide
*Safe, comprehensive removal for ANY feature toggle in FiscaalGemak codebase*

---

## 🎯 Purpose

**This single guide handles removal of ANY feature toggle from `FiscaalGemakFeatures` enum.**

Replace `{FEATURE_NAME}` with the actual feature toggle you want to remove (e.g., `ShowSheetPreview`, `MediumClassificationIsAvailable`, etc.)

---

## ⚠️ Strict Responsibility

**This guide's ONLY purpose is to REMOVE code and perform cleanup. Never add new logic, features, or conditions. Never comment out code—DELETE it entirely. Never leave comments in the codebase as a result of this process. ONLY DELETE.**

---

## 🔍 Implementation Patterns in FiscaalGemak

**Feature toggles in FiscaalGemak are implemented in TWO main ways:**

### Implementation Pattern A: Service-Based (Backend - C#)
**Used in:** Backend services, repositories, controllers

```csharp
// Injected service
private readonly IPilotFeatureService _pilotFeatureService;

// Check toggle
if (_pilotFeatureService.IsFeatureEnabled(FiscaalGemakFeatures.FeatureName))
{
    // Feature code
}
```

**What to search for:**
- `pilotFeatureService.IsFeatureEnabled`
- `_pilotFeatureService.IsFeatureEnabled`
- `IPilotFeatureService` interface

---

### Implementation Pattern B: Direct Dictionary Access (Frontend - TypeScript/Angular)
**Used in:** Angular components, services, AngularJS controllers

```typescript
// Stored in session storage
fiscaalGemakFeatures: SessionStorageData = {};

// Loaded in constructor
constructor(private sessionStorageService: FgSessionStorageService) {
    this.fiscaalGemakFeatures = this.sessionStorageService.getFeaturesStorageData();
    
    // Direct access to check toggle
    this.isFeatureEnabled = 
        this.fiscaalGemakFeatures[FiscaalGemakFeatures[FiscaalGemakFeatures.FeatureName]];
}

// Alternative direct access patterns:
if (this.fiscaalGemakFeatures[FiscaalGemakFeatures.FeatureName]) { }
if (fiscaalGemakFeatures['FeatureName']) { }
```

**What to search for:**
- `fiscaalGemakFeatures[FiscaalGemakFeatures[FiscaalGemakFeatures.`
- `this.fiscaalGemakFeatures[FiscaalGemakFeatures.`
- Variable assignments like `isFeatureEnabled = this.fiscaalGemakFeatures`
- Variable declarations like `isFeatureEnabled: boolean;`

---

### Implementation Pattern C: Stored Variable (MOST COMMON in Frontend)
**Used in:** Angular/AngularJS components, templates

```typescript
// Step 1: Variable declaration
isShowSheetPreviewEnabled: boolean;
{feature}Enabled: boolean;
{feature}Flag: boolean;
show{Feature}: boolean;

// Step 2: Assignment from dictionary
constructor() {
    this.isShowSheetPreviewEnabled = 
        this.fiscaalGemakFeatures[FiscaalGemakFeatures[FiscaalGemakFeatures.ShowSheetPreview]];
}

// Step 3: Usage in templates
@if (isShowSheetPreviewEnabled && otherCondition) { }
<div *ngIf="isShowSheetPreviewEnabled">...</div>
<div ng-if="vm.isShowSheetPreviewEnabled">...</div>
```

**⚠️ CRITICAL:** This pattern requires THREE removals:
1. **Variable declaration** → DELETE
2. **Variable assignment** → DELETE
3. **All variable usages** → DELETE or adjust conditions

**What to search for:**
- `is{Feature}Enabled`
- `{feature}Enabled`
- `{feature}Flag`
- `show{Feature}`
- Variable name in camelCase and variations

## 📖 Pattern Recognition & Removal Guide

### 🟢 Pattern 1: Simple If Block
**What it looks like:**
```csharp
if (pilotFeatureService.IsFeatureEnabled(FiscaalGemakFeatures.{FEATURE_NAME}))
{
    // New feature code
    DoSomething();
}
```

**How to remove:**
```csharp
// Remove the if wrapper, keep the contents
DoSomething();
```

---

### 🟢 Pattern 2: If-Else Block
**What it looks like:**
```csharp
if (pilotFeatureService.IsFeatureEnabled(FiscaalGemakFeatures.{FEATURE_NAME}))
{
    // New feature code (KEEP)
    DoNewThing();
}
else
{
    // Old code (DELETE)
    DoOldThing();
}
```

**How to remove:**
```csharp
// Keep only the if block, delete else
DoNewThing();
```

---

### 🟡 Pattern 3: Negated Condition (DANGER!)
**What it looks like:**
```csharp
if (!pilotFeatureService.IsFeatureEnabled(FiscaalGemakFeatures.{FEATURE_NAME}))
{
    // This runs when feature is OFF
    RemoveItem();
}
```

**How to remove:**
```csharp
// DELETE THE ENTIRE BLOCK (nothing remains)
// Feature is ON, so this code never runs
```

**⚠️ Critical:** Negated conditions mean "when feature is disabled". When removing the toggle, DELETE the entire block—don't convert to `if (false)` or keep it!

---

### 🔴 Pattern 4: Complex Boolean Logic (EXTREME DANGER!)
**What it looks like:**
```csharp
if (condition1 && 
    pilotFeatureService.IsFeatureEnabled(FiscaalGemakFeatures.{FEATURE_NAME}) && 
    condition2)
{
    DoSomething();
}
```

**How to remove:**
```csharp
// Keep other conditions, remove ONLY the toggle
if (condition1 && condition2)
{
    DoSomething();
}
```

**Another variant:**
```csharp
if (condition1 && 
    (!pilotFeatureService.IsFeatureEnabled(FiscaalGemakFeatures.{FEATURE_NAME}) || 
     condition2))
```

**How to remove:**
```csharp
// Remove the toggle, keep condition2
if (condition1 && condition2)
```

**⚠️ Critical:** 
- Preserve ALL non-toggle conditions
- Handle `&&` (AND) vs `||` (OR) logic correctly
- Watch for parentheses—preserve grouping
- Test thoroughly!

---

### 🟠 Pattern 5: Method Parameters
**What it looks like:**
```csharp
var data = repository.GetData(
    id, 
    pilotFeatureService.IsFeatureEnabled(FiscaalGemakFeatures.{FEATURE_NAME}));
```

**How to remove:**
```csharp
// Replace with true (feature is always enabled)
var data = repository.GetData(id, true);
```

---

### 🟢 Pattern 6: Ternary Operator
**What it looks like:**
```csharp

    ? newValue 
    : oldValue;
```

**How to remove:**
```csharp
// Use the enabled value (first option)
var result = newValue;
```

### 🟡 Pattern 7: Variable Assignment (Frontend - Direct Dictionary Access)
**What it looks like:**
```typescript
// Variable declaration

// Assignment in constructor/ngOnInit

    this.fiscaalGemakFeatures = this.sessionStorageService.getFeaturesStorageData();
```

**How to remove:**
```typescript
// Step 1: Delete the variable declaration
// isShowSheetPreviewEnabled: boolean; ← DELETE THIS

// Step 2: Delete the assignment
constructor(private sessionStorageService: FgSessionStorageService) {
    this.fiscaalGemakFeatures = this.sessionStorageService.getFeaturesStorageData();
    // DELETE the assignment line
}

// Step 3: Remove all usages of the variable in templates and code
```

**⚠️ Critical:** After removing the variable:
1. Search for ALL usages of the variable name (e.g., `isShowSheetPreviewEnabled`)
2. Remove from template conditions (`*ngIf`, `@if`, etc.)
3. Remove from TypeScript conditions
4. Don't replace with `true` - just remove the condition or adjust logic

### 🔵 Pattern 9: Method Signature Cleanup (Post-Removal)
**What it looks like:**
After removing a feature toggle, you might have methods that still accept a boolean parameter that's always `true`:

```csharp
// Step 1: After toggle removal, calls look like:
bool result = myObject.SomeMethod(otherParam, true);  // ← Always passing true

// Step 2: Method signature still has the parameter:
public bool SomeMethod(int otherParam, bool isFeatureEnabled)
{
    return isFeatureEnabled ? EnabledLogic() : DisabledLogic();
}
```

**How to remove (complete cleanup):**

```csharp
// Step 1: Simplify the method signature - remove the boolean parameter
public bool SomeMethod(int otherParam)  // ← Removed bool parameter
{
    // Step 2: Simplify the logic - use only the "enabled" path
    return EnabledLogic();  // ← Remove ternary, use enabled logic directly
}

// Step 3: Update all call sites - remove the boolean argument
bool result = myObject.SomeMethod(otherParam);  // ← No more true parameter
```

**Real example from ConfigureLetterheadPhase removal:**

**Before:**
```csharp
// Method signature
public static bool ApplyLetterheadForPhase(
    this StepStateType? stepState, 
    FinancialStatementStylingTemplate template, 
    bool isLetterheadFeatureEnabled)
{
    return isLetterheadFeatureEnabled 
        ? HasLetterheadForPhase(stepState, template)
        : (stepState is not null and not StepStateType.PreDraft and not StepStateType.Draft);
}

// Call site
bool result = stepState.ApplyLetterheadForPhase(template, true);

// Test
bool result = stepState.ApplyLetterheadForPhase(template, isLetterheadFeatureEnabled);
```

**After:**
```csharp
// Method signature - parameter removed, logic simplified
public static bool ApplyLetterheadForPhase(
    this StepStateType? stepState, 
    FinancialStatementStylingTemplate template)
{
    return HasLetterheadForPhase(stepState, template);  // Always use enabled path
}

// Call site - no boolean parameter
bool result = stepState.ApplyLetterheadForPhase(template);

// Test - parameter and variable removed
bool result = stepState.ApplyLetterheadForPhase(template);
```

**⚠️ Critical Steps:**
1. **Update method signature** - Remove the boolean parameter
2. **Simplify method logic** - Remove ternary/if-else, keep only the "enabled" path logic
3. **Update ALL call sites** - Remove the boolean argument from every method call
4. **Update ALL tests** - Remove boolean parameter from test method calls and delete variable declarations
5. **Verify compilation** - Ensure no remaining references to the old signature

**⚠️ When to use Pattern 9:**
- After you've completed Pattern 5 (Method Parameters) and replaced all calls with `true`
- When you notice methods that always receive `true` as a parameter
- As a final cleanup step to completely remove all traces of the feature toggle
- When the boolean parameter's sole purpose was to control feature toggle behavior

**Pro Tip:** After removing Pattern 5 occurrences, search for methods that now always receive `true` and consider removing that parameter entirely for cleaner code.

## ⚠️ Common Mistakes (DON'T DO THIS!)

| ❌ **WRONG** | ✅ **RIGHT** | Why |
|-------------|-------------|-----|
| `if (true)` | `if (condition1 && condition2)` | Preserve business logic |
| Convert `!toggle` to `if (false)` | Delete entire block | Negated means remove when enabled |
| `/* old code */` comments | Complete deletion | No comments allowed |
| Keep `IPilotFeatureService` unused | Remove from constructor | Clean up dependencies |
| Batch all changes | One pattern type at a time | Reduce risk of errors |
| `// TODO: Remove toggle` | Just remove it | No new comments |
| Add new `if (true)` checks | Direct execution | No new logic |
| Remove last `using` without blank line | Ensure only one blank line after using block | Prevents `namespaceFg...` corruption |

## 🔧 Using Statement Cleanup: Preserving Namespace Spacing

### ⚠️ The Problem
When removing the **last `using` statement** in a file (especially `using IPilotFeatureService = ...`), the whitespace/newline between the last using block and the `namespace` declaration can get corrupted:

```csharp
// ❌ WRONG - No space between using block and namespace
using IPilotFeatureService = FiscaalGemak.Core.IServices.IPilotFeatureService;

namespace FiscaalGemak.Web.Frontend.Services

// After bad cleanup becomes:
namespaceFiscaalGemak.Web.Frontend.Services  ← CORRUPTED!
```

### ✅ The Solution
When removing a `using` statement that is immediately followed by `namespace`, **preserve only ONE blank line** between them:

```csharp
// ❌ WRONG - No blank line after removing using
using OtherService;
namespace FiscaalGemak.Web.Frontend.Services

// ✅ CORRECT - Always ensure only ONE blank line after using block
using OtherService;

namespace FiscaalGemak.Web.Frontend.Services
```

### 📋 Step-by-Step for Using Statement Removal

1. **Locate the using statement to remove:**
   ```csharp
   using IPilotFeatureService = FiscaalGemak.Core.IServices.IPilotFeatureService;
   ```

2. **Check what comes after it:**
   - If followed by another `using` → Simply delete the line
   - If followed by blank line(s) then `namespace` → Simply delete the line
   - **If directly followed by `namespace` with no blank line** → Delete AND ensure only 1 blank line remains

3. **Verify the result:**
   ```csharp
   // Should look like:
   [previous using statements OR start of file]
   
   namespace FiscaalGemak.Web.Frontend.Services
   ```

### 🎯 Why This Happens
Most code cleanup tools don't intelligently handle the edge case where a `using` statement is the last one before `namespace`. They may accidentally merge the line with the namespace declaration when removing the using statement, losing the critical whitespace.

### ✅ Prevention Checklist
- [ ] After removing any `using` statement, verify there's a blank line before `namespace`
- [ ] Don't just delete the line - ensure proper spacing is maintained
- [ ] Check that both `using` blocks and `namespace` are on separate lines
- [ ] Test compilation after cleanup to catch spacing issues early

##   Troubleshooting: "I Can't Find All References!"

### Problem: Missing Frontend Variable Pattern
**If you searched for the feature name but missed variable usages:**

```typescript
// You searched for: "ShowSheetPreview"
// But you missed: "isShowSheetPreviewEnabled" ← Different name!
```

**Solution:** Search for variable name patterns:
```
is{FeatureName}Enabled
{featureName}Enabled  
{featureName}Flag
show{FeatureName}
```

**Examples:**
- `ShowSheetPreview` → search also for `isShowSheetPreviewEnabled`, `showSheetPreview`
- `MediumClassificationIsAvailable` → search for `isMediumClassificationIsAvailableEnabled`, `mediumClassificationIsAvailable`

---

### Problem: Dictionary Access Pattern Not Recognized
**If you see this pattern and don't know what to do:**

```typescript
this.someVariable = this.fiscaalGemakFeatures[FiscaalGemakFeatures[FiscaalGemakFeatures.FeatureName]];
```

**This is Pattern 7 (Variable Assignment):**
1. Delete the variable declaration
2. Delete the assignment line
3. Remove all usages of `someVariable`

---

### Problem: Template Still References Removed Variable
**Error:** `Property 'isFeatureEnabled' does not exist on type 'Component'`

**Solution:** You removed the TypeScript variable but forgot template usages:
```html
<!-- Find and remove from templates: -->
@if (isFeatureEnabled && ...)  ← Remove condition
*ngIf="isFeatureEnabled && ..."  ← Remove condition
ng-if="vm.isFeatureEnabled"  ← Remove condition
```

---

### Problem: Per-Entity Config vs Feature Toggle
**How to distinguish:**

**Feature Toggle (REMOVE):**
```typescript
// Stored in fiscaalGemakFeatures dictionary from session storage
this.isEnabled = this.fiscaalGemakFeatures[FiscaalGemakFeatures.FeatureName];
```

**Per-Entity Config (KEEP):**
```typescript
// Property on the entity/model itself
document.currentSheet.showSheetPreview  ← Keep this!
sheet.ShowSheetPreview  ← Keep this!
item.showFeature  ← Keep this!
```

**Rule:** If it comes from the `fiscaalGemakFeatures` dictionary → REMOVE
If it's a property on the entity/model → KEEP

| Variable | `var = toggle` | Delete variable | 🟡 |
| Template | `*ngIf="toggle"` | Remove condition | 🟠 |
| Dictionary Access | `fiscaalGemakFeatures[FiscaalGemakFeatures[...]]` | Pattern 7 | 🟡 |
| **Method Signature** | **`method(param, true)`** | **Remove bool param entirely** | **🔵** |
